# Clean Code — практики Роберта Мартина для TS/Vue

## Когда применять
Загружай этот skill перед написанием нетривиального кода и обязательно — при рефакторинге, ревью, разборе «запутанных» модулей. Применим к компонентам, composables, утилитам, сторам, тестам.

---

## 1. Осмысленные имена

### Должны раскрывать намерение
```ts
// ❌
const d = 5;                // что это?
function check(x: number) { return x > 0; }

// ✅
const daysSinceUpdate = 5;
function isPositive(value: number): boolean { return value > 0; }
```

### Не дезинформируй
- Не называй массив `userList`, если это `Set<User>` или `Map`. Используй `users`.
- Не используй похожие имена для разного: `XYZControllerForEfficientHandling` vs `XYZControllerForEfficientStorage`.
- `l` и `1`, `O` и `0` — недопустимы.

### Различимые имена
```ts
// ❌
function copy(a1: string[], a2: string[]) { ... }
function getActiveAccount() {}
function getActiveAccountInfo() {}   // в чём разница?

// ✅
function copy(source: string[], destination: string[]) { ... }
```

### Произносимые и поисковые
- Имена должны произноситься: `generationTimestamp` лучше `genymdhms`.
- Однобуквенные имена — только для локальных счётчиков (`i`, `j` в цикле). Всё остальное — поисковое: `MAX_PAGE_SIZE` находится `grep`-ом, `7` — нет.

### Соглашения по типу сущности
| Тип | Паттерн | Пример |
|---|---|---|
| Класс / interface типа сущности | существительное PascalCase | `UserProfile`, `BaseButtonProps` |
| Функция / метод | глагол camelCase | `fetchUsers`, `formatDate` |
| Boolean | вопрос: `is/has/can/should` | `isActive`, `hasError`, `canSubmit` |
| Константа | UPPER_SNAKE | `MAX_RETRIES`, `DEFAULT_LIMIT` |
| Обработчик события | `handleX` или `onX` | `handleClick`, `onSubmit` |
| Composable | `use{Name}` | `useDebounce`, `useScrollLock` |
| Тип-union | существительное type | `type Mode = 'mock' \| 'real'` |

### Длина имени = длина области видимости
- Локальная переменная в 3-строчной функции → можно `i`, `x`.
- Поле класса / параметр публичной функции / экспорт → полное осмысленное имя.

### Запрещены
- Кодирование типа в имя: `strName`, `arrUsers`, `iCount` (венгерская нотация).
- `data`, `info`, `temp`, `obj`, `manager`, `processor`, `helper` без уточнения.
- Числовые суффиксы: `result1`, `result2`, `data2`.

## 2. Функции

### Малые функции
- Цель: **до 20 строк**. Идеал — 4–8 строк.
- Один уровень вложенности внутри функции (максимум 2).
- Если внутри `if/else/while` — тело это **одна строка вызова другой функции**.

### Одна задача (SRP на уровне функции)
Если функцию можно описать через «и» — она делает два дела.
```ts
// ❌
async function loadAndValidateAndSaveUser(data: unknown) { ... }

// ✅
async function saveUser(data: UserInput): Promise<User> {
  const parsed = parseUserInput(data);
  validateUser(parsed);
  return persistUser(parsed);
}
```

### Один уровень абстракции
Не смешивай в одной функции высокоуровневые шаги (`fetchUsers()`) и низкоуровневые детали (`xhr.setRequestHeader('Authorization', ...)`).

### Декларативное чтение сверху вниз
Функция вызывает только функции, объявленные ниже по уровню абстракции. Сначала «что», потом «как».

### Аргументы
- **0** — идеально.
- **1** — нормально.
- **2** — допустимо.
- **3** — только при необходимости.
- **≥4** — почти всегда сигнал собрать в объект-параметр.

```ts
// ❌
createUser('Ivan', 'ivan@x.ru', 25, 'admin', true);

// ✅
createUser({ name: 'Ivan', email: 'ivan@x.ru', age: 25, role: 'admin', isActive: true });
```

### Никаких флаг-аргументов
Boolean-аргумент == функция делает два разных дела.
```ts
// ❌
render(isAdmin: boolean) {}

// ✅
renderAdmin() {}
renderUser() {}
```

### Чистые функции там, где возможно
- Одинаковый вход → одинаковый выход.
- Никаких скрытых side-effects (мутация глобалов, запись в localStorage).
- Если side-effect нужен — название обязано об этом сообщать: `saveTokenToStorage`, не `getToken`.

### Команда vs запрос (Command-Query Separation)
Функция либо **что-то делает** (команда, `void` или `Promise<void>`), либо **отвечает на вопрос** (запрос, возвращает значение). Не и то и другое.

```ts
// ❌
function set(name: string, value: string): boolean {
  if (!attrs.has(name)) return false;
  attrs.set(name, value);
  return true;
}

// ✅
if (attrs.has('name')) attrs.set('name', value);
```

### Предпочитай исключения кодам возврата
```ts
// ❌
if (deletePage(page) === E_OK) { ... }

// ✅
try {
  deletePage(page);
} catch (err) {
  logError(err);
}
```

### DRY — нет дублированию
Любой повторяющийся блок (3+ строк, 2+ раз) → выноси в функцию/composable/utility.

## 3. Комментарии

> «Комментарии — извинение за неспособность выразиться в коде» — Мартин.

### Не пиши комментарии
- Хорошее имя функции/переменной устраняет 90% комментариев.
- Если хочется комментировать — переименуй или вынеси в функцию.

```ts
// ❌
// проверка, активен ли пользователь и не забанен
if (user.flag === 1 && user.banDate === null) { ... }

// ✅
if (isActiveUser(user)) { ... }
```

### Допустимые комментарии
- **Юридические/лицензионные** — header файла.
- **TODO/FIXME** — с владельцем и тикетом: `// TODO(@ivan, #1234): кэшировать ответ`.
- **Предупреждение о последствиях** — `// ВНИМАНИЕ: запускает 5-минутный тест`.
- **Усиление намерения**, которое нельзя выразить в коде: «Используем `\r\n` потому что протокол требует CRLF».
- **JSDoc для публичного API** библиотеки/SDK.

### Запрещённые комментарии
- Журнал изменений (`// 2024-03-12 added X`) — для этого есть git.
- Закомментированный код — удаляй немедленно.
- Маркеры секций (`// ===== HELPERS =====`) — выноси в отдельный файл.
- Избыточные: `// инкремент i` перед `i++`.
- Журналы: `// Author: Ivan, Date: 2024-01-01`.

## 4. Форматирование

### Вертикальное
- Файл — как газетная статья: сверху общая идея, ниже детали.
- Связанный код держи рядом. Переменная объявляется максимально близко к первому использованию.
- Между группами импортов, между функциями — одна пустая строка.

### Горизонтальное
- Строка ≤ 120 символов.
- Пробелы вокруг операторов: `a + b`, не `a+b`.
- Без выравнивания колонок: это создаёт шум при изменениях.

### Импорты
Порядок (с пустыми строками между группами):
1. Сторонние пакеты (`vue`, `pinia`, `vee-validate`)
2. `@/` абсолютные импорты (по слоям FSD сверху вниз)
3. Относительные `./` (последним — импорт стилей)

```ts
import { defineStore } from 'pinia';
import { computed, onMounted, ref } from 'vue';

import { useUserStore } from '@/entities/user';
import { BaseButton, BaseInput } from '@/shared/ui';
import { useScopeT } from '@/shared/lib';

import { useLocalLogic } from './useLocalLogic';
import './MyComponent.scss';
```

## 5. Объекты и структуры данных

### Data class vs Object
- **Data class** (DTO, типы из API) — поля публичные, методов нет.
- **Объект (класс с поведением)** — поля приватные, методы публичные.
- Не смешивай: «гибрид» с публичными полями и публичными методами обычно — запах.

### Закон Деметры
Объект разговаривает только со своими «друзьями»:
```ts
// ❌
const path = ctx.options().getScratchDir().getAbsolutePath();

// ✅
const path = ctx.absoluteScratchPath;
```

## 6. Обработка ошибок

### Исключения, не коды
См. раздел 2. Возвращай результат, на ошибки — кидай.

### Точечный try-block
`try` тело — только то, что может бросить. Логику обработки выноси в отдельную функцию.

```ts
async function deletePage(id: string): Promise<void> {
  try {
    await pageApi.delete(id);
  } catch (err) {
    handleDeleteError(err);
  }
}
```

### Не возвращай и не передавай `null`
Возвращай:
- пустой массив `[]` вместо `null`
- объект-заглушку (`Null Object` pattern)
- бросай исключение, если значение **обязательно**

```ts
// ❌
const users = getUsers(); // вдруг null
if (users) { ... }

// ✅
const users = getUsers(); // всегда массив (возможно пустой)
if (users.length) { ... }
```

### Создавай свои типы ошибок
```ts
class NotFoundError extends Error { constructor(public id: string) { super(`Not found: ${id}`); } }
class ValidationError extends Error { constructor(public fields: string[]) { super('Validation failed'); } }
```

### Никогда не «глотай» ошибки
```ts
// ❌
try { await save(); } catch {}

// ✅
try { await save(); }
catch (err) { errorState.value = formatError(err); logError(err); }
```

## 7. Границы (Boundaries)

- Не выпускай типы сторонних библиотек сквозь публичный API своего модуля. Оборачивай.
- Учебные тесты («learning tests») — пиши перед интеграцией новой библиотеки, чтобы зафиксировать ожидаемое поведение.
- На границе с внешним миром (HTTP, FS, DB) типизация — `unknown`, дальше валидация → доменный тип.

## 8. Юнит-тесты — отдельный first-class код

См. skill `testing-standards` для деталей. Кратко:
- F.I.R.S.T: **F**ast / **I**ndependent / **R**epeatable / **S**elf-validating / **T**imely.
- Один концепт на тест (один `expect`-блок мысленно, физически — несколько связанных asserts можно).
- Та же чистота, что и в продакшн-коде: имена, малость функций, DRY (через хелперы).

## 9. Классы и модули

### Малые
- **Класс**: ≤ 5 публичных методов; ответственность описывается одним предложением без «и»/«или».
- **Vue-компонент**: `<template>` ≤ 50 строк, `<script setup>` ≤ 100 строк.
- **Файл**: ≤ 200 строк. Превышение — сигнал к декомпозиции.

### SOLID
- **S** (Single Responsibility): одна причина для изменений.
- **O** (Open/Closed): расширяемо без правки существующего кода (стратегии, плагины, slots).
- **L** (Liskov Substitution): подтип должен корректно подставляться без сюрпризов.
- **I** (Interface Segregation): мелкие специализированные интерфейсы, а не один «толстый».
- **D** (Dependency Inversion): зависим от абстракций (типов, интерфейсов), не от конкретных реализаций (HTTP-клиент → передавай как параметр).

## 10. Code Smells — главные запахи

| Запах | Симптом | Лечение |
|---|---|---|
| Длинная функция | > 20 строк | Extract Function |
| Длинный список параметров | ≥ 4 | Introduce Parameter Object |
| Дублирование | Один и тот же блок дважды | Extract Function/Composable |
| Большой класс/файл | > 200 строк | Extract Class/Module |
| Чувство ревности (Feature Envy) | Метод A постоянно дёргает поля B | Move Method to B |
| Магические числа/строки | `if (x > 7)`, `if (mode === 'a1')` | Замени на именованную константу |
| Прерывистые изменения | Одно изменение трогает много мест | Move Together (объедини близкое) |
| Дробный сдвиг (Shotgun Surgery) | Одно бизнес-изменение — N классов | Inline + Extract в один |
| Цепочки сообщений | `a.b().c().d().e()` | Hide Delegate, закон Деметры |
| Спекулятивная общность | Параметры/слои «на будущее» | YAGNI — удали |
| Мёртвый код | Не вызываемый код | Удали |
| Закомментированный код | Любой | Удали (git помнит) |
| Свич с одинаковой структурой | Повторяется в N мест | Polymorphism / map-объект |
| Загадочное имя | `data`, `manager`, `helper` | Переименуй по намерению |

## 11. Boy Scout Rule

> Оставь код чище, чем нашёл.

Заметил неиспользуемый импорт, дублирование, плохое имя рядом со своим изменением — поправь. Но **не переписывай чужой код без запроса**, если это вне минимальной правки.

---

## 12. YAGNI и KISS

- **YAGNI**: не добавляй функциональность «на всякий случай». Только то, что нужно сейчас.
- **KISS**: при выборе между двумя решениями — простое побеждает. Сложная абстракция оправдана только реальной потребностью.

## Чеклист Clean Code перед коммитом

- [ ] Каждое имя раскрывает намерение
- [ ] Нет сокращений и магических значений
- [ ] Функции ≤ 20 строк
- [ ] Аргументов ≤ 3 или объект-параметр
- [ ] Нет флаг-аргументов
- [ ] Один уровень абстракции на функцию
- [ ] Нет дублирования (DRY)
- [ ] Нет комментариев, объясняющих «что» (вместо них — хорошие имена)
- [ ] TODO с владельцем и тикетом
- [ ] Закомментированный код удалён
- [ ] Импорты сгруппированы и упорядочены
- [ ] Ошибки не «проглатываются»
- [ ] Не возвращается `null` без явной причины
- [ ] Закон Деметры не нарушен (`a.b.c.d` подозрителен)
- [ ] Файл ≤ 200 строк, класс — одна ответственность
- [ ] Применены SOLID-принципы где это уместно
- [ ] Не оставлен мёртвый/спекулятивный код
- [ ] Правило бойскаута: код вокруг стал чище
