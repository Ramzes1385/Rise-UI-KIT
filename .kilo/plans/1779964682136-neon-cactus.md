# План: Аудит тестов + 100% покрытия storybook/components, unit composables/utils

> Цель: проверить текущее состояние тестов, зафиксировать coverage компонентов (storybook play-runner), composables (vitest unit) и utils (vitest unit), составить детальный план доведения до 100%.

## 🔴 FIX: merge-coverage самоперезапись storybook-источника (2026-05-28)

### Симптом
`coverage/index.html` показывает composables/utils (useDebounce, useCustomStyle, formatUtils, useIcon, navigationUtils, rangeUtils) как **0%**, хотя `test:composables:coverage` и `test:utils:coverage` дают по ним **100%**.

### Причина
- `getCoverageDir()` для `COVERAGE_MODE=storybook` возвращает `'coverage'` → storybook пишет `coverage/coverage-final.json`.
- `scripts/merge-coverage.mjs` читает `coverage/coverage-final.json` как 4-й источник И пишет туда же merged-данные (`json` reporter + `MERGED_DIR='coverage'`).
- Итог: после первого merge storybook-источник **перезаписан** merged-данными. Повторный merge читает сам себя; если unit-json'ы отсутствуют — отчёт деградирует до «storybook-only», где composables/utils = 0% (play-runner их не вызывает).

### Необходимые правки в коде (заблокированы правами edit, применить вручную)

**`build/tests/vitest.config.ts` — `getCoverageDir()`:**
```ts
function getCoverageDir(): string {
	if (COVERAGE_MODE === 'storybook') return 'coverage/storybook' // было 'coverage'
	if (COVERAGE_MODE === 'components') return 'coverage/components'
	return 'coverage'
}
```

**`scripts/merge-coverage.mjs` — SOURCES[3]:**
```js
const SOURCES = [
	'coverage/components/coverage-final.json',
	'coverage/composables/coverage-final.json',
	'coverage/utils/coverage-final.json',
	'coverage/storybook/coverage-final.json', // было 'coverage/coverage-final.json'
]
const MERGED_DIR = 'coverage'
```

После этих правок storybook пишет в `coverage/storybook/`, а merge пишет в `coverage/` — самоперезаписи нет, повторные прогоны стабильны.

### Корректная процедура без правок (обходной путь)
1. `npm run test:utils:coverage` → `coverage/utils/coverage-final.json`
2. `npm run test:composables:coverage` → `coverage/composables/coverage-final.json`
3. `npm run test:components:coverage` → `coverage/components/coverage-final.json`
4. `npm run test:storybook:coverage` → `coverage/coverage-final.json`
5. `Copy-Item coverage/coverage-final.json coverage/storybook/coverage-final.json` (сохранить storybook ДО merge)
6. `node scripts/merge-coverage.mjs` (читает 4 источника, пишет в `coverage/`)
7. **Перед повторным циклом**: удалить `coverage/coverage-final.json` и перезапустить storybook (иначе merge прочитает merged-данные).

## A. Контекст тестовой инфраструктуры

### A.1 Vitest projects (`build/tests/vitest.config.ts:137-174`)

В Vitest сконфигурированы **два проекта**:

| Проект | Окружение | Команда |
|---|---|---|
| `unit-integration` | `jsdom`, pool `forks` | `npm run test:unit`, `test:integration`, `test:vitest` |
| `storybook` | Browser (Playwright Chromium) через `@storybook/addon-vitest` | `npm run test:storybook`, `test:storybook:coverage` |

Тесты компонентов (`*.spec.ts`/`*.integration.spec.ts` рядом с `Base*.vue`) могут идти в обоих проектах.
Тесты composables (`src/composables/**/*.spec.ts`) и utils (`src/utils/**/*.spec.ts`) идут только в `unit-integration` (jsdom).

### A.2 Coverage режимы (env `COVERAGE_MODE`)

В `vitest.config.ts:67-80` действует логика:

```ts
function getCoverageInclude(): string[] {
  if (isComponentCoverage()) return COMPONENT_COVERAGE_INCLUDE  // ['src/components/**/*.vue']
  return BASE_COVERAGE_INCLUDE                                  // ['src/utils/**/*.ts', 'src/composables/**/*.ts', 'src/components/**/*.vue']
}

function isComponentCoverage(): boolean {
  return COVERAGE_MODE === 'components' || COVERAGE_MODE === 'storybook'
}
```

Это значит:
- `COVERAGE_MODE=storybook` → отчёт **только по `.vue`** (composables/utils выпадают).
- `COVERAGE_MODE=components` → то же самое + `--coverage.thresholds.100 --coverage.thresholds.perFile`.
- БЕЗ `COVERAGE_MODE` (npm run `test:vitest`) → отчёт по utils + composables + components.vue.


Замечу: `test:components:coverage` запускает **`unit-integration`-проект** (jsdom), а не storybook browser. Имя обманчиво.

### 2.1 `build/config/css.ts` — Sass modern-compiler + строка

**Файл:** `build/config/css.ts:34-53`

**Было** (плохо для скорости):

```ts
// addGlobalImports — функция, перекомпилирует переменные на каждый файл
return {
  preprocessorOptions: {
    scss: {
      additionalData: addGlobalImports,
      silenceDeprecations: ['legacy-js-api'],
      charset: false,
    },
  },
  modules: { localsConvention: 'camelCaseOnly', generateScopedName: ... },
  devSourcemap: !isProduction,
}
```

**Станет** (modern-compiler + строка):

```ts
const GLOBAL_SCSS_PRELUDE = [
  '@use "@styles/variables" as *;',
  '@use "@styles/mixins" as *;',
  '@use "@styles/functions" as *;',
  '',
].join('\n')

export function createCssConfig(mode: string): CSSOptions {
  const isProduction = isProductionMode(mode)

  return {
    preprocessorOptions: {
      scss: {
        api: 'modern-compiler',
        additionalData: GLOBAL_SCSS_PRELUDE,
        charset: false,
      },
    },
    modules: {
      localsConvention: 'camelCaseOnly',
      generateScopedName: isProduction ? '[hash:base64:8]' : '[name]__[local]--[hash:base64:5]',
    },
    devSourcemap: !isProduction,
    transformer: 'lightningcss', // см. § 2.3
  } as CSSOptions
}
```

Удалить вспомогательные функции: `addGlobalImports`, `shouldSkipImport`, `normalizeSource`, `addGlobalImports`, константы `GLOBAL_SCSS_IMPORTS`, `GLOBAL_SCSS_FILES`.

**Тонкость с самоимпортом:** ранее функция проверяла `shouldSkipImport(filename)`, чтобы `_variables.scss` не импортировал сам себя. С `additionalData` как строкой Sass корректно работает с `@use … as *;`: повторный `@use` файла, который уже находится в графе, игнорируется. Однако в **самих файлах** `_variables.scss`, `_mixins.scss`, `_functions.scss` (`src/styles/`) такой prelude бы создал self-reference. Vite решает это так: `additionalData` применяется **только к содержимому файла, поступившему в препроцессор**, то есть к `*.scss` файлам, которые импортируются через `<style lang="scss">` в Vue-компонентах. К файлам, которые компилируются как часть `@use` из них — НЕ применяется (это transitive resolution в Sass-компиляторе, без participate Vite). **Поэтому строка-prelude безопасна** для нашего набора `_variables/_mixins/_functions` — они resolve'ятся через @use из preludе, а не как entry-файлы.

Но если бы где-то в проекте был `*.scss` файл-entry внутри `src/styles/`, который содержит непосредственно определения переменных и не использует prelude — проблем не было бы. Текущая проверка `shouldSkipImport` в `css.ts:19-21` была обоснована тем, что `additionalData` КАК ФУНКЦИЯ вызывается на каждый компилируемый файл, включая партиалы при первом запуске; modern-compiler API ведёт себя иначе и эту защиту не требует.

## 6. Риски и митигация

### 6.1 Lightning CSS строже постcss

**Риск:** дубликаты селекторов, нестандартный CSS могут вылезти.
**Митигация:** если падает — удалить `transformer: 'lightningcss'`, остаться на дефолтном минификаторе.

### 6.2 Sass modern-compiler меняет поведение `@use`

**Риск:** разный resolution path для `@use "@styles/variables"`.
**Митигация:** modern-compiler **полностью** совместим со стандартом `@use`, а alias `@styles` идёт через Vite-resolver (это не legacy `importer`). Проблем не ожидается, но если возникнет — добавить `loadPaths` или `importers` в SCSS-опции.

### 6.3 Oxc minify даёт другой output

**Риск:** main chunk +/-5 KB, иной mangling, может всплыть редкий рантайм-эдж-кейс (например, eval-зависимые библиотеки).
**Митигация:** прогоны unit + visual тестов. Если что-то падает — откат на terser, но с упрощёнными опциями (см. § 2.4).

### 6.4 Удаление `pinia`/`vue-router` ломает что-то скрытое

**Риск:** Storybook-аддоны или плагины могут ожидать их в peer.
**Митигация:** грепом убедиться, что они нигде не упомянуты, кроме `package.json`. Если Storybook ругнётся — вернуть как peerDependencies или optionalDependencies.

### 6.5 Удаление `terser`/`obfuscator` ломает Vitest или Storybook

**Риск:** в transitive они могут быть нужны другим dev-tool.
**Митигация:** после `npm uninstall` запустить `npm install` и затем `npm run test:unit`, `npm run storybook` — убедиться, что нет ошибок типа `Cannot find module 'terser'`.

### 6.6 `modulePreload.polyfill: false` ломает старый Safari

**Риск:** Safari ≤ 16.3 не понимает `<link rel="modulepreload">`.
**Митигация:** наш `target: 'es2020'` уже отсекает Safari ≤ 13.1. Для Safari 14–16.3 (где `es2020` есть, но modulepreload нет) — preload просто игнорируется, без падения. Контент всё равно подгрузится через обычный `<script type="module">`.

### 6.7 Sprite в `dist/` через `emitFile` не работает в dev

**Риск:** dev-сервер не отдаёт сгенерированный SVG.
**Митигация:** Шаг 9 включает `configureServer` middleware. Если возникнут проблемы — оставить **Вариант B** (запись в `public/icons.svg` только при изменении содержимого) — этот вариант гарантированно работает и убирает шум в git.
